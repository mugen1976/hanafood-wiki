# 파싱 파이프라인

카카오톡 발주방 메시지 → DB `발주` 테이블 저장까지의 전체 데이터 흐름.

## 전체 흐름

```
카카오톡 메시지
     │
     ▼
① on_chat() — 메시지 수신 (handlers.py)
     │
     ├─ 관리방 → _handle_admin_room() (명령어 처리)
     └─ 발주방 → _handle_order_room()
                    │
                    ▼
② OrderParser.parse() — 텍스트 파싱
     ├─ _preprocess(): "추가요"/"취소해주세요" 등 접미사 제거
     ├─ _parse_with_heuristic(): 정규식 기반 휴리스틱 (우선)
     └─ _parse_with_gemini(): Gemini fallback (복합 패턴)
     결과: list[ParsedItem(name, qty, unit)]
                    │
                    ▼
③ Matcher.resolve() — 품목 매칭
     ├─ DB find_product_candidates(매출처, 품목명)
     ├─ _score_candidate(): 점수 산정 (품명1=정확 > 단위일치)
     └─ 결과: (resolved[], ambiguous[], missing[])
                    │
                    ▼
④ 확정 (auto_confirm / 수동 Y)
     ├─ 자동: resolved만 있고 ambiguous/missing 없음 → 바로 INSERT
     ├─ 수동: 관리방 미리보기 → 운영자가 Y → INSERT
     └─ 예외: ambiguous/missing 있으면 예외함 등록 → 수동 처리
                    │
                    ▼
⑤ DB insert_orders_bulk() — 발주 테이블 저장
     INSERT INTO 발주 (매출처, 품명1, 품명2, 수량, 단위, 매입처, 비고, 발주일, 품명번호, 확인, 대기업코드)
                    │
                    ▼
⑥ 발주발송 파이프라인 (선택)
     ├─ !발주발송 → 미리보기 → !발송확인 → 매입처별 이미지/엑셀 전송
     └─ 자동발송: AUTO_DELIVERY_TIME 도달 시 자동 실행
```

## ① 메시지 수신 — on_chat()

**파일:** `handlers.py`

```python
def on_chat(self, ctx) -> None:
    text = self._normalize_incoming_text(ctx.message.msg)
    # 1. 엑셀 첨부 검사 (_handle_excel_attachment)
    # 2. 관리방 여부 확인
    # 3. 발주방 → _handle_order_room()
```

- **관리방** → `_handle_admin_room()`: `Y`/`N` 확정, `!등록`, `!발송확인` 등 명령어 처리
- **발주방** → `_handle_order_room()`: 주문 메시지라고 판단되면 파싱 시작

**발주방 등록:** 운영자가 `!이방등록 <매출처명>` 으로 room_id ↔ 매출처 연결.  
`.env`의 `ADMIN_USER_IDS`에 포함된 사용자만 등록 가능.

### 메시지 전처리

```python
_normalize_incoming_text(text):
    - NFKC 정규화 (전각 문자 → 반각)
    - ZWSP/ZWJ/BOM 제거
```

## ② 텍스트 파싱 — OrderParser

**파일:** `parser.py`  
**출력 타입:** `ParsedItem(name: str, qty: float, unit: str)` (parsed_types.py)

### 처리 순서 (중요)

```
message
  → _preprocess(): "취소해주세요" / "추가요" / "추가해주세요" 접미사 제거
  → _parse_with_heuristic(): 정규식 파싱 (1차 시도)
     └─ 성공하면 즉시 반환 (빠르고 결정적)
  → _parse_with_gemini(): Gemini 2.5 Flash fallback (2차 시도)
     └─ 휴리스틱 실패 시에만 호출
```

### 휴리스틱 파서 (`_parse_with_heuristic`)

**규칙 기반**, 결정적(deterministic), 금액 단위 정확히 처리.

```python
# 1. 공백이 있으면 split → 각 token을 _parse_token()
# 2. 공백이 없으면 compact regex로 분할
_parse_token(token):
    품목명 + 수량 + (단위) 패턴 추출
    - 정육 패턴: "정육20" → name="정육", qty=20
    - 금액 단위: "콩나물1만원" → name="콩나물(1만원)", qty=1, unit="봉"
    - 금액 숫자(>=1000): "콩나물15000" → name="콩나물(15000원)", qty=1, unit="봉"
    - 일반: "양파1망" → name="양파", qty=1, unit="망"
```

**정규화 규칙 (`_normalize_items_units`):**
- `k` → `kg`
- `키로` → `kg`
- `박스` 유지 (box로 변환하지 않음)
- `개` 유지

### Gemini 파서 (`_parse_with_gemini`)

**모델:** Gemini 2.5 Flash (google-genai SDK)  
**출력:** JSON (response_mime_type="application/json", schema 제약)

```python
prompt = f"""
다음 한국어 주문 텍스트를 JSON으로 파싱하세요.
출력은 스키마에 맞는 JSON 한 개만 반환하세요.

규칙:
- 주문이 아니면 items는 빈 배열.
- 품명수량단위, 품명수량, 공백 없는 연속 패턴을 모두 처리.
- 단위가 없으면 unit은 빈 문자열.

{wiki_content}  # ← 위키 주문-패턴.md 참조

입력:
{text}
"""
```

**Gemini 호출 조건:** 휴리스틱이 결과를 반환하지 못한 경우에만.  
**Gemini 장점:** 정정 메시지("물미역이아니고미역줄기요"), 복합 패턴 처리 가능.

### 위키 연동

`OrderParser._load_all_wiki()`에서 다음 위키 페이지들을 로드:
- `/root/wiki/concepts/주문-패턴.md` — 발주처별 공통 주문 패턴
- `/root/wiki/concepts/파싱-파이프라인.md` — 전체 파이프라인 구조 (본 문서)
- `/root/wiki/entities/*.md` — 발주처 개별 entity 페이지 (아리식당, 가영식당, 태종대요양병원, 윌리스요양병원 등)

YAML frontmatter 제거 후 본문만 `self._wiki_content`에 캐싱.  
각 페이지는 `## 주문 패턴`, `## 파싱 파이프라인`, `## 발주처: {이름}` 헤더와 함께 Gemini prompt의 `{wiki_section}`으로 삽입됨.

```python
def _load_all_wiki(self, wiki_path: str | None) -> None:
    parts: list[str] = []
    # concepts 페이지
    parts.append("## 주문 패턴 ...\n{주문-패턴.md}")
    parts.append("## 파싱 파이프라인 ...\n{파싱-파이프라인.md}")
    # entity 페이지
    for ef in sorted(glob.glob("/root/wiki/entities/*.md")):
        name = ef.rsplit("/", 1)[-1].rsplit(".", 1)[0]
        parts.append(f"## 발주처: {name}\n{content}")
    self._wiki_content = "\n\n".join(parts)
```

> Gemini는 concepts(공통 패턴) + entities(매출처별 상세 패턴)을 함께 참고하여
> 발주처 특화 패턴(아리식당 붙여쓰기, 가영식당 띄어쓰기, 병원 엑셀 등)을 정확히 파싱 가능.

## ③ 품목 매칭 — Matcher

**파일:** `matcher.py`  
**출력 타입:**
- `ResolvedOrder(requested: ParsedItem, product: dict, converted_qty: float)` — 단일 매칭
- `AmbiguousOrder(requested: ParsedItem, candidates: list)` — 후보 다수
- `missing: list[ParsedItem]` — 매칭 실패

### 매칭 과정

```python
for each ParsedItem:
    candidates = DB.find_product_candidates(매출처, 품목명)
    if no candidates → missing에 추가
    score = _score_candidate(item, candidate)
        - 품명1 == item.name → +6점 (정확 일치)
        - 발주단위 == item.unit → +3점
        - 단위 == item.unit → +2점
    if unique best → ResolvedOrder
    if tied top-2 → AmbiguousOrder
```

**발주규격 변환:** `convert_qty()` — 품목의 `발주규격` 필드에 `*2`나 `/2` 같은 변환식이 있으면 수량 조정.

### DB 조회 (`find_product_candidates`)

```python
# db.py — MSSQL 프로시저 호출
EXEC ? = dbo.USP_BOS_S_PROD_SRCH ?, ?, ?, ?, 1, 1000
# 매출처명과 검색어로 품목 후보 반환
```

## ④ 확정 — auto_confirm / 수동

### 자동 확정 (AUTO_CONFIRM_ORDERS = true)

```python
_handle_order_room():
    resolved, ambiguous, missing = matcher.resolve()
    if resolved and not ambiguous and not missing:
        # 모든 품목이 단일 매칭 → 자동 INSERT
        rows = _insert_rows_from_resolved(...)
        insert_ids = db.insert_orders_bulk(rows)
        관리방에 "N건 자동 등록됨" 알림
    elif parse_failed or ambiguous or missing:
        # 예외함에 등록 → 운영자 수동 처리 대기
```

**비고=추가 자동 설정:**  
`AUTO_CONFIRM_EXTRA_MEMO_AFTER=16:00` — 오후 4시 이후 자동확정 시 비고에 "추가" 저장.

### 수동 확정 (관리방)

```
관리방 미리보기 예시:

  [아리식당] 2026-05-18
  · 양파 1망 → 동양종합식품
  · 대파 2단 → 동양종합식품
  · 배추 1통 → 감자/대관령상회

  Y / N (선택: !선택 1,3 / !선택 2-4)
```

- `Y` → 전품목 확정 → `insert_orders_bulk()`
- `N` → 취소
- `!선택 1,3,5` → 선택 품목만 확정, 나머지는 예외함

## ⑤ DB 저장 — 발주 테이블

**파일:** `db.py` — `insert_orders_bulk(rows)`

```sql
INSERT INTO 발주
    (매출처, 품명1, 품명2, 수량, 단위, 매입처, 비고, 발주일, 품명번호, 확인, 대기업코드)
    OUTPUT INSERTED.번호
VALUES (?, ?, ?, ?, ?, ?, ?, ?, ?, 0, ?)
```

### 필드 설명

| 필드 | 출처 | 설명 |
|------|------|------|
| `매출처` | 채팅방 등록명 | 발주방 이름 (아리식당, 가영식당 등) |
| `품명1` | DB 매칭 결과 | `find_product_candidates`의 품명1 |
| `품명2` | ParsedItem.name | 사용자가 입력한 품목명 (그대로) |
| `수량` | ParsedItem.qty (+/- 변환) | 발주규격 반영 후 수량 |
| `단위` | ParsedItem.unit | kg, 박스, 봉, 망, 판, 개 등 |
| `매입처` | DB 매칭 결과 | 공급처 (동양종합식품, 감자/대관령상회 등) |
| `비고` | auto/memo | "추가" or 빈 문자열 |
| `발주일` | parse | 주문이 적용될 배송일 |
| `품명번호` | DB 매칭 결과 | 제품 고유 ID |
| `확인` | 고정 | 0 (미확인 상태) |
| `대기업코드` | DB 매칭 결과 | 대기업 코드 (보통 빈 문자열) |

### 품명1 vs 품명2

- **품명1:** DB에서 검색된 정규화된 품목명 (매칭 기준)
- **품명2:** 사용자가 원문에 쓴 그대로의 품목명 (보존)

예: "양파(국산)1망" → 품명1="양파(국산)", 품명2="양파(국산)"

## ⑥ 취소 처리

**파일:** `handlers.py` — `_handle_order_room_cancel()`, `db.py` — `cancel_order_quantities()`

```python
취소 메시지 → parser.parse_with_cancel_flag()
    → (items, was_cancel=True)
    → cancel_order_quantities(발주일, 매출처, [(품명번호, 차감수량)])
```

**차감 방식:** 같은 품명번호가 여러 행이면 `번호` 오름차순으로 먼저 차감.  
차감 후 수량이 0이 되면 행 DELETE. 차감량이 남으면 "부족" 보고.

관련 문서: [[영수증-검증]] — 매입처 영수증 OCR·단가 검증 시스템

## 데이터 구조 요약

```
ParsedItem (parser 출력)
 ├── name: str     — 품목명 (정규화 전)
 ├── qty: float    — 수량
 └── unit: str     — 단위 (kg, 박스, 봉...)

ResolvedOrder (매칭 성공)
 ├── requested: ParsedItem
 ├── product: dict     — DB 품목 정보 (품명1, 품명번호, 매입처, 발주단위, 발주규격)
 └── converted_qty: float  — 발주규격 반영 후 수량

AmbiguousOrder (매칭 모호)
 ├── requested: ParsedItem
 └── candidates: list  — 상위 5개 후보

DB 발주 테이블 행
 ├── 매출처, 품명1, 품명2, 수량, 단위
 ├── 매입처, 비고, 발주일, 품명번호
 ├── 단가, 금액 (영수증 검증 시 저장)
 └── 확인=0, 대기업코드
```
